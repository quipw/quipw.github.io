Style Rules

You should be using #1, 2 , 4, 5, 6, 7, 8 from the first submission. The others will make sense as the quarter progresses.

  1. Put a comment at the top of your code that tells what the application does and includes a @author tag containing your name. Use this style comment  /** .... */
  2. Class name starts with an upper case letter. Each subsequent word starts with upper case.

    Example: HelloWorld

  3. Other identifiers (method names, variable names, object names) start with lower case and use Camel Casing

    Example: thisIsCamelCasing

  4. Align the opening and closing braces of all blocks of code  under the first letter in the statement

    public static void main(String[ ] args)
    {
       //statements
    }
  5. Do not put any other statement on the line with the opening {
  6. Indent each block of code one more level. A block is the code between the braces { and }
  7. Indent at least 3 and no more than 4 spaces.Don't use tabs. They can cause misalignment of the code when viewed in a different application You lose points for improper alignment
  8. Do not have tabs in your code. Replace tabs with spaces. See the module Tabs: Removing and Preventing
  9. Use meaningful variable names. Meaningful variable names give information about what the variable is for. Do not use single letters (except for loop control.)  Do not leave out vowels. Do not use abbreviations.Call the variable to hold the cost of an object cost.

    Example: "cost" rather than "c" or cst

  10. Use named constants in your program when you need a literal other than 0, 1, or 2. Constant names should be all upper case. Give the constant a name that describes its function. Don't name a constant SEVEN. Call it DAYS_PER_WEEK.

    Example: TAX_RATE

  11. Do not do calculations in an output statement. Do the calculation in another statement(s) and assign the value to a variable.  Then use the variable in the output statement.

    GOOD:
    totalCost= quantity * price;
    System.out.println("The total cost is "+ totalCost);

    BAD:
    System.out.println("The total cost is "+ (quantity + price));

  12. Do not print "raw" numbers. Tell what the number is. For example:

    GOOD:
    The area of a circle with radius 1 is 3.14

    BAD:
    3.14
  13. Put spaces around operators. Computers do not care, but humans do.

    GOOD
    cost = price + price * TAX_RATE;

    BAD
    cost=price+price*TAX_RATE;

  14. Break your code into logical sections with blank lines. Logical sections might be input, calculations and outputs. But do not put a blank line between every line of code.
  15. When a statement is very long, break it into multiple lines. A line is too long if the reader has to scroll to see all of the statement. Limit line size to about 80 characters. Indent subsequent lines beyond the start of the first
  16. A method should have only one return statement. It is acceptable in a very short method to have two returns. Or to use a return at the top of a method to handle an error.
  17. Instance variable must be private
  18. Every public method except the main should be documented with javadoc notation.
  19. Summary of rules for classes
    • Start class names with an uppercase letter
    • Put a javadoc style comment (/** ... */ at the top of the class telling what the class does. Include your name in @author tag
    • Put a javadoc comment above every public method telling what the method does. Include @param tags to describe the parameters and @return to tell what the method returns
    • Always make instance variables private
